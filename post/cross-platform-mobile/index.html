Cross Platform Mobile Musings

June 30, 2015

Mobile has taken over the world. Everyone is embracing mobile as a space they must either start in or occupy.

But what does this means for developers and businesses trying to create a competitive product. Here are my thoughts on the matter. How I would proceed with my own products.

Product Requirements

iOS App

  • Communication either RESTful or realtime with the server
  • Local storage of data
  • Native UI
  • Elegant design and fantastic user experience
  • Test suite unit, UI, and end to end integration tests

Android App

  • Communication either RESTful or realtime with the server
  • Local storage of data
  • Native UI
  • Elegant design and fantastic user experience
  • Test suite unit, UI, and end to end integration tests

Server

  • Stores and secures data
  • Provides authentication (Facebook, Google +, and native to your product. These are at a minimum)
  • Easy rapid scalability
  • RESTful API
  • Realtime API whether its publication/subscription or synchronization (Not a hard requirement yet but it will be in the next year or two as websockets take over the world)
  • Front End UI to be accessed from the desktop browser and the mobile browser
  • Test suite unit, UI, and end to end integration tests

Strategies

Phew and thats just to begin with! If you choose to drop either Android, iOS, or the browser based interface you’ll alienate a large percentage of your user population. The Android/iOS split is arguably 50⁄50 and the percentage of people that will refuse to download your app and instead use your web site is pretty large as well. I don’t know about you but as I read the list I just see $$$$$$$$$ going off in my head as well as the technical debt down the road.

Above all, put your time and money into what makes the most sense for your business. Not what the new hotness is.

No Money But Want To Reach The Most Users

Build the server and web UI first. Ignore native mobile and cross platform mobile. Work on a solid backend server and a fast but simple UI. That UI should still be responsive and easily usable on even small mobile devices. Stay away from too much css trickery and JavaScript magic.

You’ll spend your energy building a backend that can eventually be used across all clients. And, the web UI will reach the largest number of users since the web is platform agnostic.

Once the product starts bringing in some money, start investing in native mobile and desktop clients. Don’t hesitate to waste time on cross platform apps thats what the mobile web UI is for. No point in packaging it up and wasting memory on a user’s phone.

Note: There are some products that require the app to be native. Whether they need to take advantage of GPS, camera, video, UI responsiveness, graphics, local storage, etc. At that point, native apps should come first. Server side may need to be something like firebase or the open sourced version of Parse. Whatever takes the least amount of time to bring the product to its base level of functionality.

Doesn’t Require a Backend Server

Very few products fall into this category, but if they do go to town on building amazing native UI’s. Spend the time crafting the best user experience.

Enough Money to Pour Down the Drain

Build everything server, web UI, mobile UI, native mobile, and native desktop. But this scenario rarely ever happens.
